为什么要在 Emacs 里做 GTD

Getting Things Done（GTD）是 David Allen 提出的一套时间管理方法论，其核心思想是：把脑中所有待办事项全部"清空"到一个可信赖的外部系统中，然后按流程收集、厘清、组织、回顾、执行。这套方法论的价值在于，它让大脑从"记住事情"的负担中解脱出来，专注于"做事情"本身。

对于 Emacs 用户而言，Org Mode 天然就是这个"可信赖的外部系统"。纯文本、可版本控制、可编程——没有什么比 .org 文件更让人安心了。社区中也早有成熟的 GTD 实现：org-gtd.el，它在 Org Mode 之上封装了完整的 GTD 工作流，从收集（Capture）到厘清（Clarify）再到执行（Engage），一气呵成。

然而，当你的使用场景是"公司一台 MacBook Pro，家里一台 Mac Mini，二者共享同一套 ~/.emacs.d 配置"时，一个关键问题浮出水面：**GTD 数据如何在两台设备之间实时同步？**

我的解决方案是一套两层自动同步机制：第一层负责实时的 commit & push，第二层作为安全网定时清理和兜底。整套配置写在 init-gtd.el 中，下面逐层展开。

org-gtd 基础配置

首先是 org-gtd 本身的配置，这一部分比较直白：

(use-package org-gtd
  :ensure t
  :after org
  :demand t
  :custom
  (org-gtd-update-ack "4.0.0")
  (org-gtd-directory "~/gtd")
  (org-todo-keywords '((sequence "TODO" "NEXT" "WAIT" "|" "DONE" "CNCL")))
  (org-gtd-keyword-mapping '((todo . "TODO")
                             (next . "NEXT")
                             (wait . "WAIT")
                             (done . "DONE")
                             (canceled . "CNCL")))
  :config
  (org-edna-mode 1)
  (setq org-agenda-files (list org-gtd-directory))
  :bind
  (("C-c d c" . org-gtd-capture)
   ("C-c d e" . org-gtd-engage)
   ("C-c d p" . org-gtd-process-inbox)
   ("C-c d n" . org-gtd-show-all-next)
   ("C-c d s" . org-gtd-reflect-stuck-projects)
   :map org-gtd-clarify-mode-map
   ("C-c c" . org-gtd-organize)
   :map org-agenda-mode-map
   ("C-c ." . org-gtd-agenda-transient)))

几个要点：

  • *GTD 目录*：所有 GTD 数据集中存放在 ~/gtd ，这个目录本身是一个 git 仓库
  • *TODO 关键字*：定义了 TODO → NEXT → WAIT 的流转序列，以及 DONE / CNCL 两种终态。这对应了 GTD 中"下一步行动"（Next Action）和"等待"（Waiting For）的概念
  • *org-edna*：一个 Org Mode 插件，允许在任务之间定义依赖关系。比如"任务 A 完成后自动将任务 B 标记为 NEXT"，这在管理项目型任务时非常实用
快捷键 功能 GTD 阶段
C-c d c 快速收集想法到 inbox 收集
C-c d p 逐条厘清 inbox 中的条目 厘清
C-c d e 查看可执行的任务列表 执行
C-c d n 显示所有"下一步行动" 执行
C-c d s 回顾卡住的项目 回顾

日常使用中的典型流程是：脑中闪过一个念头，=C-c d c= 快速记录到 inbox；积攒了几条后，=C-c d p= 逐条厘清——是可执行的任务、某个项目的一部分、还是仅供参考的资料？厘清时按 C-c c 调用 org-gtd 的组织面板来分类。最后 C-c d e 进入 engage 视图，专注执行。

第一层：autosync-magit 实时同步

~/gtd 是一个 git 仓库，同步的本质就是 git 操作。autosync-magit 是一个基于 Magit 的小型 minor mode，专门为"用 git 在多设备间同步个人内容"而设计。它的工作原理很简单：

  1. 保存文件时，自动 git add + =git commit=（带 debounce）
  2. 定时执行 git pull
  3. commit 后延迟执行 git push

配置如下：

(use-package autosync-magit
  :ensure nil
  :load-path "site-lisp/autosync-magit"
  :commands autosync-magit-mode
  :custom
  (autosync-magit-commit-message "chore(gtd): auto-sync")
  (autosync-magit-pull-timer 600)        ; 每 10 分钟 pull 一次
  (autosync-magit-push-debounce 600)     ; commit 后最多等 10 分钟再 push
  (autosync-magit-pull-interval 10))     ; 任意两次 pull 之间至少间隔 10 秒

通过 .dir-locals.el 激活

autosync-magit 的激活方式很优雅——不需要在 init 文件中硬编码路径，而是在 ~/gtd/.dir-locals.el 中声明：

((nil . ((autosync-magit-commit-message . "chore(gtd): auto-sync")
         (autosync-magit-pull-timer . 600)
         (autosync-magit-push-debounce . 600)
         (mode . autosync-magit))))

这意味着：当你在 Emacs 中打开 ~/gtd/ 下的任何文件时，autosync-magit-mode 会自动启用，同步机制随即生效。关闭所有 ~/gtd/ 下的 buffer 后，同步自然停止。这种"按需激活"的设计非常 Emacs。

为了让 Emacs 信任这些目录局部变量而不弹出确认提示，需要在 init 文件中将它们标记为安全：

(dolist (pair '((autosync-magit-commit-message . stringp)
                (autosync-magit-pull-timer      . numberp)
                (autosync-magit-push-debounce   . numberp)))
  (put (car pair) 'safe-local-variable (cdr pair)))
(add-to-list 'safe-local-eval-forms '(autosync-magit-mode 1))

用 rebase 替代 merge

autosync-magit 原生的 pull 策略是 fetch + merge 。但在我的双设备场景下，第二层同步（稍后会讲到）会在主设备上做 squash + force-push，这意味着远端历史会被改写。此时 merge 会产生大量无意义的合并提交。

因此，我通过 define-advice 将 pull 策略替换为 fetch + rebase ：

(define-advice autosync-magit-pull
    (:override (path) rebase-and-reset-on-conflict)
  "Fetch + rebase; on conflict, abort and reset to upstream."
  ;; ... 获取仓库路径，执行 fetch ...
  ;; fetch 完成后的 sentinel 中：
  (when (not (magit-rev-ancestor-p "@{upstream}" "HEAD"))
    (let ((exit-code (process-file "git" nil nil nil "rebase" "@{upstream}")))
      (cond
       ((zerop exit-code)
        (run-hooks 'autosync-magit-after-merge-hook))
       (t
        ;; rebase 冲突时：中止 rebase，创建备份分支，然后硬重置到上游
        (process-file "git" nil nil nil "rebase" "--abort")
        (gtd--create-backup-branch)
        (process-file "git" nil nil nil "reset" "--hard" "@{upstream}")
        (gtd--revert-repo-buffers repo-dir))))))

冲突处理策略是 “以远端为准” ——如果 rebase 遇到冲突，先创建一个备份分支（形如 backup/gtd-20260313-143022-a1b2 ）保存本地状态，然后 reset --hard 到上游版本。这看起来激进，但对于 GTD 数据来说是合理的：两台设备上不太可能同时编辑同一行，真正发生冲突时，保留远端版本并保留本地备份，比让同步卡住要好得多。

第二层：Safety-net 定时安全网

第一层解决了"Emacs 运行时"的同步问题。但如果 Emacs 没有运行呢？或者 autosync-magit 因为网络问题 push 失败了呢？再者，一天下来几十个 chore(gtd): auto-sync 的提交，git log 看起来也不太美观。

第二层 Safety-net 就是为了解决这些问题。它基于 elcron（一个 Emacs 内的 cron 调度器），每天定时执行三次。

主设备与副设备的分工

两台设备的 Safety-net 行为不同：

设备 角色 定时时间 行为
公司 secondary 11:45, 17:45, 22:45 plain commit + push
家里 primary 11:50, 17:50, 22:50 squash 当天提交 + AI 生成 commit

副设备比主设备早 5 分钟运行。这样当主设备的 Safety-net 触发时，副设备今天的提交已经 push 到远端了，主设备 fetch 后可以一并 squash。

设备角色的判定通过 macOS 的 scutil --get LocalHostName 实现：

(defun gtd-safety-net--primary-p ()
  "Return non-nil if the current device is the primary host."
  (string= (string-trim (shell-command-to-string "scutil --get LocalHostName"))
           gtd-safety-net-primary-host))  ; "Dis-Mac-mini"

由于两台设备使用同一套 ~/.emacs.d 配置，这个运行时判定让同一份代码在不同设备上自动执行不同的逻辑。

副设备：plain commit + push

副设备的逻辑相对简单：

  1. 如果有未提交的改动，=git add -A= + =git commit=（带 AI 生成的 commit message）
  2. 如果有未推送的提交（包括 autosync-magit 提交成功但 push 失败的情况），=fetch= + rebase + push

这是一张兜底的安全网：即使 autosync-magit 没有正常工作，一天至少有三次机会把改动推上去。

主设备：squash + force-push

主设备的行为更有意思。它会将当天所有提交 squash 成一个提交，并用 AI 生成一条有意义的 commit message。

流程如下：

  1. 先提交所有未暂存的改动
  2. fetch + rebase 整合远端提交
  3. 找到"今天第一个提交之前的那个提交"作为 reset 目标
  4. git reset --soft 到该目标（保留所有改动在暂存区）
  5. 用 git diff 获取今天的全部变更
  6. 调用 Claude Code 根据 diff 生成 commit message
  7. 用生成的 message 创建一个新提交
  8. git push --force-with-lease 推送到远端
# squash 前的 git log
a1b2c3d chore(gtd): auto-sync     ← 今天的自动提交
e4f5g6h chore(gtd): auto-sync
i7j8k9l chore(gtd): auto-sync
m0n1o2p [2026-03-12] 完成周报，整理读书笔记  ← 昨天的 squash 结果

# squash 后的 git log
x1y2z3w [2026-03-13] 新增季度 OKR 草案，调整项目优先级  ← AI 生成
m0n1o2p [2026-03-12] 完成周报，整理读书笔记

Claude Code 生成 commit message

这是整套方案中最有趣的部分。Safety-net 会调用 Claude Code 的 --print 模式（非交互式，直接输出结果），将 git diff 作为输入，让 AI 生成一条结构化的 commit message。

Prompt 的要求是：

  • 第一行格式为 [时间] 变更要点
  • 空行后用列表逐项说明
  • 不加 Conventional Commits 前缀
(defcustom gtd-safety-net-claude-binary "/usr/local/bin/mc"
  "Absolute path to the Claude Code binary."
  :type 'file)

(defcustom gtd-safety-net-claude-timeout 120
  "Timeout in seconds for Claude Code AI calls."
  :type 'natnum)

(defcustom gtd-safety-net-max-diff-size (* 100 1024)
  "Maximum diff size in characters to send to Claude."
  :type 'natnum)

这里用绝对路径 /usr/local/bin/mc 指定 Claude Code 二进制文件，是因为 GUI Emacs 的 PATH 环境变量往往不完整。超时设为 120 秒，diff 超过 100KB 会被截断——毕竟 GTD 数据的单日变更通常不会太大。

如果 Claude Code 调用失败（超时、网络问题等），会 fallback 到一个朴素的时间戳 commit message：

chore(gtd): auto-commit 2026-03-13 17:50

幂等性设计

主设备的 squash 操作是幂等的。每次 squash 后的提交会附带一个 git trailer：

Squashed-by: gtd-safety-net

下次 Safety-net 运行时，会检查 HEAD 提交是否已包含这个 trailer。如果包含，说明今天已经 squash 过了，跳过即可。这意味着即使 Safety-net 一天触发三次，squash 也只会执行一次。

elcron 调度

elcron 在 Emacs 内部实现了类似 cron 的调度功能。它的 cron 表达式是 7 字段的：

秒 分 时 日 月 星期 年

Safety-net 的调度在 elcron 加载后自动启动：

(with-eval-after-load 'elcron
  (gtd-safety-net-start))

手动管理也很方便：=M-x gtd-safety-net-start= 启动，=M-x gtd-safety-net-cancel= 取消，=M-x gtd-safety-net-commit= 立即执行一次。

两层联动：全局同步锁

两层同步共享同一个 Emacs 进程，因此需要协调。如果 Layer 1 正在 push 的同时 Layer 2 开始 squash，结果将不可预测。

解决方案是一个全局互斥锁 gtd--sync-lock ：

(defvar gtd--sync-lock nil
  "Non-nil when a GTD sync operation is in progress.")

(defmacro gtd--with-sync-lock (&rest body)
  "Execute BODY if the sync lock is available, skip otherwise."
  `(if gtd--sync-lock
       (message "[gtd] Sync already in progress, skipping.")
     (unwind-protect
         (progn (setq gtd--sync-lock t) ,@body)
       (setq gtd--sync-lock nil))))

Layer 2 通过 gtd--with-sync-lock 包裹整个 commit 流程；Layer 1 的 pull 和 push 也会检查这个锁。任何时刻只有一层在执行 git 操作，另一层会优雅地跳过。

备份与清理

冲突处理时创建的备份分支不会无限累积。Safety-net 每次运行结束后会清理超过 30 天的备份分支：

(defcustom gtd-safety-net-backup-retention-days 30
  "Number of days to retain backup branches."
  :type 'natnum)

小结

这套方案的核心思想是 “让同步完全透明” ：打开 GTD 文件、记录想法、保存、关闭——同步在背后自动发生。两层机制互为补充：

  • *Layer 1*：实时性好，但依赖 Emacs 运行且依赖网络
  • *Layer 2*：定时兜底，保证数据终究会同步；同时在主设备上压缩历史，保持 git log 的整洁

对于"公司 + 家里"的双 Mac 场景，这套方案运行了一段时间后，体验非常流畅——几乎不需要操心同步问题，专注于 GTD 本身即可。
